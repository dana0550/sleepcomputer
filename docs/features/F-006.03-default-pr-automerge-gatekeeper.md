# [F-006.03] Default PR Auto-Merge Gatekeeper

## Summary

Automatically merge pull requests to `main` by default only after all checks succeed, cursor bugbot is successful, and bugbot review threads are resolved.

## Goals

- Remove manual "enable auto-merge" button usage for routine PRs.
- Keep CI/check gates strict before merge execution.
- Enforce cursor bugbot pass + no unresolved bugbot review threads before merge.

## Non-Goals

- Bypassing branch protection or required status checks.
- Requiring human review approvals for merge readiness.

## Requirements

- R1: `pr-automerge` workflow must evaluate PRs targeting `main` and merge automatically when all gates are clear.
- R2: Auto-merge gating must require all check runs and commit status contexts on the PR head SHA to be completed and successful (or neutral/skipped for check-run conclusions where applicable).
- R3: Auto-merge gating must require a successful cursor bugbot signal and zero unresolved bugbot-authored review threads.
- R4: Repository must provide a per-PR opt-out control that disables default auto-merge without requiring code changes.
- R5: Workflow must re-evaluate merge readiness on PR lifecycle events and on periodic schedule so merges proceed after late-arriving checks/comments resolve.

## Acceptance Criteria

- AC1: New eligible PRs to `main` merge automatically once checks and bugbot gates pass.
- AC2: PRs with unresolved bugbot review threads remain unmerged until threads are resolved and bugbot checks are successful.
- AC3: Checking `Disable default auto-merge for this PR` in the PR template blocks auto-merge for that PR.
- AC4: Applying `auto-merge:off` label blocks auto-merge for that PR.
- AC5: Auto-merge resumes for opt-in PRs when CI completes successfully without requiring a human to click the auto-merge button.
